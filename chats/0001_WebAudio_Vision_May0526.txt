The "perfect pattern" you are remembering is officially known as Perfect Negotiation. It was introduced by the W3C WebRTC Working Group specifically to cure the exact trauma you experienced: state mismatch deadlocks, racial asynchronous timing issues, and session description protocol (SDP) collisions (traditionally called "glare"). 


In classic WebRTC, you had to explicitly write one code block for the "caller" and an entirely separate, asymmetric code block for the "callee". If both peers tried to dynamically change the state or modify data channels at the same time, the state machine would fracture completely. 


The beauty of the Perfect Negotiation pattern is that the exact same code runs on both peers, making it entirely symmetric. It smoothly handles collisions under the hood by splitting the peers into two abstract roles: one is Polite and the other is Impolite. 


The Core Mechanics: Politeness & Rollbacks


Politeness has absolutely nothing to do with who initiated the initial network connection. It is simply an arbitrary tie-breaker boolean flag (polite = true/false) assigned during the initial handshake. It handles collisions gracefully: 


The Impolite Peer (polite = false): Stays stubborn. If it sends an out-of-band audio offer, and an incoming offer from the other user arrives at the exact same millisecond, it aggressively ignores the incoming offer and demands that its own offer be processed. 


The Polite Peer (polite = true): Stays flexible. If a collision occurs, it instantly executes an SDP Rollback ({type: 'rollback'}). It essentially says, "Never mind my changes for a second, let me clear my local state queue, process your incoming modifications first, and then I will re-queue my changes right after we achieve stability." 


Integrating Perfect Negotiation into your NetNode Subclass 


Because your system uses a dedicated NetNode file system abstraction, this pattern fits flawlessly into the node's internal state engine. You can encapsulate the entire boilerplate inside your system's network provider, completely shielding your application code (S:/apps/MyMusicApp.js) from ever knowing it exists.


The architecture relies on tying three core WebRTC mechanisms directly together: 


1. The onnegotiationneeded Event Trigger


Whenever a developer's app interacts with a NetNode in a way that modifies the underlying pipeline (e.g., calling a method to open a secondary real-time audio channel, or requesting an ICE network restart because a user switched from Wi-Fi to cellular data), the native peer connection automatically fires an onnegotiationneeded event. 


Your NetNode internal handler catches this event and immediately generates an SDP offer, shipping it out over your system's signaling channel (S:/coms/net-signaling.js). The app developer didn't have to trigger this manually; the browser's engine handled the event scheduling. 


2. The Collision Shield


When an incoming message arrives from the signaling server, the NetNode inspects its internal signaling state. If a collision is detected (i.e., we sent an offer, but we just received an offer too): 


The Impolite Node ignores the collision entirely.


The Polite Node stops execution, applies the native rollback description to reset its local state back to stable, and then gracefully accepts the remote peer's settings. 


3. Automatic Regeneration 


Once the polite node yields and processes the impolite node's state updates, the local connection returns to a stable state. Because the polite node still has unfulfilled local modifications that it originally wanted to make, the native engine notices the variance and automatically fires onnegotiationneeded a second time. This smoothly processes the polite peer's changes right afterward, with zero data loss or synchronization deadlocks.

## Deep Technical Specification: Filesystem Polymorphism & WebRTC Fabric

### 1. Unified Filesystem Node Polymorphism (`FSNode` Inheritance)

All filesystem interactions in LOTW are bound to the `String.prototype.toNode()` contract, which returns a promise resolving to an instance of a specialized subclass of the abstract `FSNode` parent class. This architecture isolates application logic from storage and network implementations via strict method overriding.

#### Subclass Architectural Blueprints:


- **`DirNode` (IndexedDB Namespace Mapping):**


  - Manages directory structures, hierarchies, and path resolutions.


  - Interrogates IndexedDB tables to track metadata and locate child pointers.


  - Overrides child enumeration methods to return arrays of dynamic `FSNode` references when commands like `ls` query them.


- **`FileNode` (OPFS Block-Level Driver):**


  - Maps virtual user-facing file paths directly to raw binary contents stored within `OPFS:/blobs/`.


  - Content references inside IndexedDB are stored as opaque UIDs.


  - Overrides `.read()` and `.write()` methods, interacting directly with browser file-handle locks underneath the UI context.


- **`NetNode` (WebRTC DataChannel Streaming Provider):**


  - Bound to paths prefixed with the virtual `/net/` namespace. It does not hit local disks or databases.


  - Acts either as a dynamic folder (e.g., `/net/peers/`) or an active network pipe/stream (e.g., `/net/peers/alice_99/stream`).


  - Overrides `.write(bytes)`: Intercepts raw byte buffers and instantly forwards them over an active WebRTC DataChannel connection. If the connection is uninitialized or closed, it transparently pauses the returned Promise, kicks off the network handshake, and flushes data upon channel readiness.


  - Overrides `.read()` / Stream Attaching: Functions as an event-driven buffer. Application layers subscribing to a stream endpoint on a `NetNode` are registered directly to WebRTC data reception events, entirely shifting incoming UDP network packets into application-space byte inputs.


### 2. Symmetrical Connection Layer: WebRTC Perfect Negotiation


To prevent asynchronous state deadlocks, race conditions, and SDP collisions ("glare") inside the `NetNode` network provider, connection pipelines strictly execute the **W3C Perfect Negotiation** pattern. This ensures that identical, symmetric code blocks run on all communicating clients without caller/callee variance.


#### Core Execution Protocols:


- **Politeness Allocation:** During session discovery or room assignment via the signaling wire, a singular boolean flag (`polite = true / false`) is arbitrated and assigned to each peer as a definitive tie-breaker.


- **The Impolite Node Handler (`polite = false`):** Operates under absolute authority. If an incoming remote connection offer collides with an unfulfilled local state change, the impolite node completely ignores the remote offer and forces its own signaling states to process.


- **The Polite Node Handler (`polite = true`):** Operates under absolute flexibility. If a signaling collision occurs while waiting for stability, the polite node halts its current promise cascade, invokes a native WebRTC SDP Rollback (`{type: 'rollback'}`), clears its local state queue to return to stable status, processes the incoming remote adjustments first, and automatically fires `onnegotiationneeded` a second time to execute its original queued operations once the lane is clear.


- **Autonomous Triggers:** All network state shifts, stream additions, or ICE restarts natively invoke the `onnegotiationneeded` hook. The `NetNode` captures this automatically, completely insulating indie application code from the signaling lifecycle.
